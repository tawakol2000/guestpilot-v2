# Data Model: Autopilot Shadow Mode

All changes are additive. New columns on existing tables are nullable with safe defaults so the feature can be retired cleanly by dropping the columns and the new table.

## 1. `Message` — extended

New nullable columns on the existing `Message` model. All normal (non-preview) messages leave these columns null, so existing queries and rendering continue unchanged.

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `previewState` | `PreviewState?` (new enum) | yes | `null` for normal sent/received messages. See enum below for preview lifecycle. |
| `originalAiText` | `String? @db.Text` | yes | Populated when `previewState` is/was `PREVIEW_PENDING`. Holds the exact AI-generated text at the moment the preview was created, so the tuning analyzer can diff it against `content` if the admin edits before Send. |
| `editedByUserId` | `String?` | yes | Set at Send time if `content` differs from `originalAiText`. Used by the analyzer and for audit. References `User.id` (no foreign key enforced; `User` model is nominal in this codebase). |
| `aiApiLogId` | `String?` | yes | Foreign key reference to `AiApiLog.id` for messages generated by the AI pipeline. Lets the tuning analyzer load the full generation context (tool calls, SOPs consulted, FAQ entries) without heuristic matching. |

**New enum `PreviewState`** (Prisma enum in `schema.prisma`):

```prisma
enum PreviewState {
  PREVIEW_PENDING   // Unsent preview; actionable IFF it is the latest preview on the conversation
  PREVIEW_LOCKED    // Unsent preview that has been superseded by a newer preview
  PREVIEW_SENDING   // Brief transition state during the Send endpoint; prevents double-send
}
```

**State transitions**:

```text
[no previewState] → AI generates normally, sent to guest, final state

[no previewState] → shadow mode intercepts → PREVIEW_PENDING
PREVIEW_PENDING   → new preview generated on same conversation → PREVIEW_LOCKED (terminal)
PREVIEW_PENDING   → admin clicks Send (atomic UPDATE) → PREVIEW_SENDING
PREVIEW_SENDING   → Hostaway delivery succeeds → [no previewState]   (becomes a normal AI message)
PREVIEW_SENDING   → Hostaway delivery fails → PREVIEW_PENDING         (retry allowed)
```

**Invariants**:
- Only the most recent `PREVIEW_PENDING` message per conversation is eligible for Send. The Send endpoint enforces this by re-querying.
- Transitioning from `PREVIEW_PENDING` to `PREVIEW_SENDING` is a conditional UPDATE (`WHERE id = ? AND previewState = 'PREVIEW_PENDING'`). Zero rows affected ⇒ 409 Conflict.
- On successful Send, `previewState` becomes null and `hostawayMessageId` is filled, making the row indistinguishable from a normal sent AI message in downstream code except for the preserved `originalAiText` / `editedByUserId` audit metadata.

**Indexing**: an index on `(conversationId, previewState)` to keep the "lock older pending previews" UPDATE fast:
```prisma
@@index([conversationId, previewState])
```

## 2. `TenantAiConfig` — extended

One new column:

| Column | Type | Default | Notes |
|---|---|---|---|
| `shadowModeEnabled` | `Boolean` | `false` | When true, the ai.service **copilot** branch renders replies as in-chat preview bubbles instead of the legacy suggestion-card flow, and the tuning analyzer fires on edited sends. **Autopilot is not affected in either state.** Read via the existing cached `tenantConfig` service (60s cache). |

No other `TenantAiConfig` changes.

## 3. `TuningSuggestion` — new model

```prisma
model TuningSuggestion {
  id                   String               @id @default(cuid())
  tenantId             String
  sourceMessageId      String               // Preview Message that triggered the analyzer
  sourceAiApiLogId     String?              // Convenience — mirrors sourceMessage.aiApiLogId at creation time
  actionType           TuningActionType
  status               TuningSuggestionStatus @default(PENDING)
  rationale            String               @db.Text

  // EDIT action payload fields (nullable, populated per actionType)
  beforeText           String?              @db.Text  // Current content of target artifact (EDIT_*)
  proposedText         String?              @db.Text  // Replacement text (EDIT_*) OR proposed new content (CREATE_SOP)

  // Target reference fields (nullable, populated per actionType)
  systemPromptVariant  String?              // 'coordinator' | 'screening'        (EDIT_SYSTEM_PROMPT)
  sopCategory          String?              // e.g. 'sop-checkin'                 (EDIT_SOP_CONTENT / EDIT_SOP_ROUTING / CREATE_SOP)
  sopStatus            String?              // 'DEFAULT'|'INQUIRY'|'CONFIRMED'|'CHECKED_IN'  (EDIT_SOP_CONTENT / CREATE_SOP)
  sopPropertyId        String?              // Present only when AI consulted a property override (EDIT_SOP_CONTENT)
  sopToolDescription   String?              // New tool description text (EDIT_SOP_ROUTING) OR proposed for CREATE_SOP
  faqEntryId           String?              // Existing FAQ entry id              (EDIT_FAQ)

  // CREATE_FAQ payload
  faqCategory          String?
  faqScope             String?              // 'GLOBAL' | 'PROPERTY'
  faqPropertyId        String?              // When scope = PROPERTY
  faqQuestion          String?              @db.Text
  faqAnswer            String?              @db.Text

  // Final applied payload (after optional edit-then-accept)
  appliedPayload       Json?                // The payload actually written to the target on Accept. For EDIT actions: { text: "..." }. For CREATE_FAQ: { question, answer }. For CREATE_SOP: { content, toolDescription }. May differ from proposedText / faqQuestion / faqAnswer if the admin edited before accepting. Null until accepted.
  appliedAt            DateTime?
  appliedByUserId      String?

  createdAt            DateTime             @default(now())
  updatedAt            DateTime             @updatedAt

  tenant               Tenant               @relation(fields: [tenantId], references: [id], onDelete: Cascade)
  sourceMessage        Message              @relation(fields: [sourceMessageId], references: [id], onDelete: Cascade)

  @@index([tenantId, status, createdAt])
  @@index([sourceMessageId])
}

enum TuningActionType {
  EDIT_SYSTEM_PROMPT
  EDIT_SOP_CONTENT
  EDIT_SOP_ROUTING
  EDIT_FAQ
  CREATE_SOP
  CREATE_FAQ
}

enum TuningSuggestionStatus {
  PENDING
  ACCEPTED
  REJECTED
}
```

**Required-field matrix per `actionType`**:

| actionType | Required fields |
|---|---|
| `EDIT_SYSTEM_PROMPT` | `systemPromptVariant`, `beforeText`, `proposedText`, `rationale` |
| `EDIT_SOP_CONTENT` | `sopCategory`, `sopStatus`, `beforeText`, `proposedText`, `rationale` (plus `sopPropertyId` iff the AI used an override) |
| `EDIT_SOP_ROUTING` | `sopCategory`, `beforeText` (current toolDescription), `sopToolDescription` (proposed), `rationale` |
| `EDIT_FAQ` | `faqEntryId`, `beforeText`, `proposedText`, `rationale` |
| `CREATE_SOP` | `sopCategory`, `sopStatus`, `sopToolDescription`, `proposedText` (content), `rationale` (plus optional `sopPropertyId`) |
| `CREATE_FAQ` | `faqCategory`, `faqScope`, `faqQuestion`, `faqAnswer`, `rationale` (plus `faqPropertyId` iff scope=PROPERTY) |

Validation happens at two layers:
1. OpenAI `json_schema` enforces the discriminated union at generation time (§IV).
2. The `TuningSuggestion` create controller re-validates before insert (defense in depth against direct API abuse).

**Grouping by source preview**: the frontend lists suggestions ordered by `createdAt DESC` but groups them by `sourceMessageId` so multiple suggestions produced by a single analyzer run appear together. No DB-level grouping field is needed.

## 4. `SopDefinition`, `SopVariant`, `SopPropertyOverride`, `FaqEntry` — unchanged

The Accept action writes directly to these existing tables using their existing controllers. No schema changes. `systemPromptCoordinator` / `systemPromptScreening` on `TenantAiConfig` are also written directly; the existing `systemPromptHistory` JSON field captures the before-snapshot so accept-history is queryable without any new audit table.

---

## Migration

Single `prisma db push` applies all changes. Because every new column is nullable or has a default, **no data migration is needed**. Existing rows in `Message` and `TenantAiConfig` pick up the new columns with null / false values. `TuningSuggestion` starts empty.

Retirement migration is symmetric: one `ALTER TABLE` per new column + a `DROP TABLE TuningSuggestion` + `DROP TYPE` for the two enums + `PreviewState`. No data rescue needed (the whole point of the feature is a tuning period whose output is the accepted changes applied directly to existing artifacts).
