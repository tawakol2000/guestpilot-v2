# Research: Perfect AI Mix

## R1: Schema field ordering in OpenAI strict JSON schema

**Decision**: Keep old schema exactly (4 fields coordinator, 2 fields screening). No reasoning field.

**Rationale**: The OpenAI Responses API `reasoning` parameter (effort: low/medium) produces hidden reasoning tokens BEFORE output generation. These influence the output without competing for output tokens or adding schema complexity. Adding a `reasoning` field to the schema would pay twice for chain-of-thought while degrading guest_message quality through field competition. The API parameter is already in use via `pickReasoningEffort`.

**Alternatives considered**:
- Reasoning as FIRST field (v4 approach): Adds 30-80 output tokens, risks truncation, creates field competition. Rejected.
- Reasoning as LAST field (initial mix proposal): Cheaper than first-field, but still unnecessary given API parameter. Rejected.
- No reasoning at all: Current approach. API parameter provides reasoning for quality. Debugging visibility comes from stored API response IDs and AI logs.

## R2: Screening state tracking — model self-report vs. code detection

**Decision**: New `screening-state.service.ts` that scans conversation history with regex patterns for nationality and composition mentions, checks open tasks for existing screening decisions, and computes a phase (GATHER/DECIDE/POST_DECISION).

**Rationale**: Model self-report booleans (nationality_known, composition_known) were unreliable — the model contradicted itself across turns. Code-based detection is deterministic. The regex approach has a small false-negative risk (misses unusual nationality mentions) but the failure mode is benign (one redundant ask).

**Alternatives considered**:
- Model self-report booleans (v4): Unreliable, caused re-asking. Rejected.
- No tracking at all (old): Causes re-asking and re-screening. Rejected.
- NLP-based extraction (named entity recognition): Over-engineered for this use case. Simple regex with nationality/composition word lists covers 95%+ of cases. Rejected.

## R3: History window size — 10 vs 20 messages

**Decision**: 10 messages + conversation summary (v4 approach).

**Rationale**: The old system used 20 raw messages. V4 reduced to 10 but added summary injection (generated by gpt-5-nano after each response when >10 messages). The summary captures critical context (identity, complaints, decisions) more efficiently than raw messages. 10 recent + summary costs fewer tokens while providing better long-conversation context.

**Alternatives considered**:
- 20 messages, no summary (old): Loses deep context on long stays. Higher token cost. Rejected.
- 20 messages + summary: Summary would overlap with messages 11-20, creating duplicate context. Rejected.
- 10 messages + summary (v4): Best balance. Chosen.

## R4: SOP migration strategy — delete+reseed vs. targeted migration

**Decision**: Targeted migration script that checks for v4 multi-path markers (`<paths>`, `<sop>`, `Path A:`) before updating. Only update SOPs whose content still matches the v4 seed data. Preserve all user-customized content.

**Rationale**: Users may have customized SOP content through the SOPs management page. Delete+reseed would destroy those customizations. The migration script compares current DB content against known v4 seed data and only updates matching entries.

**Alternatives considered**:
- Delete all + reseed (initial proposal): Destroys user customizations. Rejected.
- No migration, manual reset: Requires user to manually reset each SOP. Poor UX. Rejected.
- Targeted migration with v4 content fingerprinting: Chosen.

## R5: search_available_properties scope for screening

**Decision**: Remove INQUIRY and PENDING from the tool's `agentScope` in the database seed and via migration script.

**Rationale**: The screening agent's job is to screen, not search properties. Premature search (before nationality is known) wastes 50+ seconds and returns meaningless results. If a guest asks "what do you have?", the property-info SOP and FAQ cover it. If they want alternatives, escalate as `property-switch-request`.

**Alternatives considered**:
- Keep tool but add description gate: Model might ignore the gate (as observed). Rejected.
- Keep tool but only enable after DECIDE phase: No mechanism to toggle tool availability mid-conversation. Rejected.
- Remove from INQUIRY/PENDING scope: Clean, deterministic. Chosen.

## R6: Escalation note format — prompt vs. schema description

**Decision**: Put a lightweight format guide in the schema `note` field description, not in the prompt body. Free-form with example pattern.

**Rationale**: Schema descriptions are read during constrained generation — the model sees them when filling the field. This saves prompt tokens while placing the instruction where it's most effective. The description includes an example pattern rather than rigid field separators, allowing natural notes.

**Schema description for coordinator `escalation.note`**: "Details for manager: situation, what the guest wants (quote their words when charged), and suggested action. Example: 'AC not working for 2 hours, guest says it's unbearable — recommend dispatching technician now.'"

**Schema description for screening `manager.note`**: "Details for manager: guest nationality, party composition, screening recommendation. Example: 'Jordanian married couple from Amman, 2 guests. Recommend acceptance.'"

## R7: get_sop auto-enrich removal

**Decision**: Remove auto-enrich from get_sop handler. Let the model chain tools instead.

**Rationale**: Auto-enrich was a workaround for the old system's limitation (schema enforcement during tool rounds blocked tool chaining). With v4's deferred schema enforcement, the model can call get_sop then check_extend_availability in sequence. Explicit tool chaining is more transparent and lets the model reason about the availability data.

**Prompt addition**: Add to coordinator `<tools>` section: "For early check-in or late checkout requests, call get_sop first, then call check_extend_availability to check for back-to-back bookings."

## R8: Reasoning visibility without schema field

**Decision**: Use a combination of: (1) the API reasoning parameter for quality, (2) AI API logs for debugging, (3) a derived one-line summary from the response structure for the inbox UI.

**Rationale**: The `showAiReasoning` toggle needs something to display. Without a schema field, we derive a summary: "Answered from SOP: sop-cleaning" or "Escalated: ac-not-working (immediate)" or "Screening: eligible-arab-family-pending-docs". This is computed from ragContext (tool calls, escalation, manager data) — not from the model output.

**For the inbox UI**: `reasoning` field in SSE broadcast is populated from ragContext, not from the model's JSON output. Format: "[action]: [detail from tool/escalation]".

## R9: Language instruction — one line

**Decision**: Replace "Always English" with "Match the guest's language. Default Egyptian Arabic if Arabic. Notes in English."

**Rationale**: GPT-5.4-mini handles Egyptian Arabic, Arabizi, and code-switching natively. It needs permission (removing "Always English") and one dialect anchor ("Egyptian Arabic"). No code-level language detection needed. The model naturally uses حضرتك for formal first contact without instruction.

14 words. No register instructions. No dialect exclusions. No Arabizi handling.

## R10: WiFi SOP content gap

**Decision**: Add WiFi troubleshooting content to the CHECKED_IN status variant only. Keep DEFAULT empty. Keep INQUIRY variant as informational only.

**Rationale**: WiFi troubleshooting (restart router, check band, etc.) only applies to guests physically in the property (CHECKED_IN). Adding to DEFAULT would serve troubleshooting to cancelled/unknown-status guests. INQUIRY guests should only hear "WiFi available, details after check-in."

**CHECKED_IN variant addition**: "If guest reports WiFi issues, share credentials again. If still not working, escalate immediately — connectivity issues require prompt resolution."
